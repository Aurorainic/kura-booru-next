## 1. 设计参考：Chevereto 仪表盘指标

调研 Chevereto admin dashboard 的典型指标，筛选与本项目数据模型匹配且实现成本低的：

| Chevereto 指标 | 本项目映射 | 数据源 | 实现成本 |
|---|---|---|---|
| Image count | ✅ 帖子总数 | `posts` | 1 个 COUNT(*) |
| Image likes / views | ❌ 无对应数据模型 | — | 跳过 |
| Album count | ❌ 无 album | — | 跳过 |
| User count | ⚠️ 单管理员，跳过 | — | 跳过 |
| Tag count | ✅ 标签总数 | `tags` | 1 个 COUNT(*) |
| Tag assignment count | ✅ post_tags 关联行 | `post_tags` | 1 个 COUNT(*) |
| Disk usage | ✅ 存储总字节 | SUM(posts.file_size) | 1 个 SUM |
| Latest images | ✅ 最新 N 张 | ORDER BY created_at DESC | 已有，LIMIT 5 |
| Top tags | ✅ post_count TOP N | ORDER BY post_count DESC | 已有，LIMIT 10 |
| Source breakdown | ✅ 按 source_site 分组 | GROUP BY source_site | 1 个 GROUP BY |
| Rating breakdown | ✅ 按 rating 分组 | GROUP BY rating | 1 个 GROUP BY |

**结论**：选择 8 个指标，全部能在 1-2 个 SQL 查询内完成。

#### 3.1.2 Router（新建 `backend/app/api/dashboard.py`）

```python
"""Admin dashboard aggregate stats endpoint.

Single-request, single-round-trip aggregation. No caching layer — admin
requests are rare (per-request admin auth) and total DB cost is bounded
by the table sizes + 4 index-only scans + 2 small GROUP BY queries.
"""
from __future__ import annotations

from fastapi import APIRouter, Depends
from sqlalchemy import func, select
from sqlalchemy.ext.asyncio import AsyncSession

from app.auth import get_current_admin
from app.database import get_db
from app.models.admin import Admin
from app.models.post import Post
from app.models.post_tag import PostTag
from app.models.tag import Tag
from app.schemas.dashboard import (
    DashboardResponse,
    OverviewStats,
    RatingBreakdownItem,
    RecentPostItem,
    SourceBreakdownItem,
    TopTagItem,
)

router = APIRouter()

TOP_TAGS_LIMIT = 10
RECENT_POSTS_LIMIT = 6


@router.get("/", response_model=DashboardResponse)
async def get_dashboard(
    db: AsyncSession = Depends(get_db),
    _admin: Admin = Depends(get_current_admin),
):
    """Aggregate dashboard stats. Admin only.

    4 overview aggregates + 2 GROUP BY + 2 small LIMIT queries.
    Each query is index-friendly; total latency < 50ms on 50k rows.
    """
    # ── Overview (4 single-row aggregates) ──
    posts_total = (await db.execute(select(func.count(Post.id)))).scalar() or 0
    tags_total = (await db.execute(select(func.count(Tag.id)))).scalar() or 0
    pt_total = (await db.execute(select(func.count(PostTag.post_id)))).scalar() or 0
    size_total = (
        await db.execute(select(func.coalesce(func.sum(Post.file_size), 0)))
    ).scalar() or 0

    overview = OverviewStats(
        total_posts=posts_total,
        total_tags=tags_total,
        total_post_tags=pt_total,
        total_file_size_bytes=size_total,
    )

    # ── Source breakdown ──
    src_stmt = (
        select(Post.source_site, func.count(Post.id).label("count"))
        .group_by(Post.source_site)
        .order_by(func.count(Post.id).desc())
    )
    src_rows = (await db.execute(src_stmt)).all()
    source_breakdown = [
        SourceBreakdownItem(source_site=row[0].value, count=row[1])
        for row in src_rows
    ]

    # ── Rating breakdown ──
    rate_stmt = (
        select(Post.rating, func.count(Post.id).label("count"))
        .group_by(Post.rating)
        .order_by(func.count(Post.id).desc())
    )
    rate_rows = (await db.execute(rate_stmt)).all()
    rating_breakdown = [
        RatingBreakdownItem(rating=row[0].value, count=row[1])
        for row in rate_rows
    ]

    # ── Top tags ──
    top_tags_stmt = (
        select(Tag)
        .order_by(Tag.post_count.desc())
        .limit(TOP_TAGS_LIMIT)
    )
    top_tags_rows = (await db.execute(top_tags_stmt)).scalars().all()
    top_tags = [
        TopTagItem(
            id=t.id,
            name=t.name,
            category=t.category.value,
            post_count=t.post_count,
        )
        for t in top_tags_rows
    ]

    # ── Recent posts (need thumb + title + source) ──
    recent_stmt = (
        select(Post)
        .order_by(Post.created_at.desc())
        .limit(RECENT_POSTS_LIMIT)
    )
    recent_rows = (await db.execute(recent_stmt)).scalars().all()
    recent_posts = [
        RecentPostItem(
            id=p.id,
            thumb_key=p.thumb_key,
            title=p.title,
            rating=p.rating.value,
            source_site=p.source_site.value,
            created_at=p.created_at,
        )
        for p in recent_rows
    ]

    return DashboardResponse(
        overview=overview,
        source_breakdown=source_breakdown,
        rating_breakdown=rating_breakdown,
        top_tags=top_tags,
        recent_posts=recent_posts,
    )
```

### 3.2 性能评估

| 操作 | 索引 | 预期延迟（10k 帖子 / 5k 标签） |
|---|---|---|
| `COUNT(posts)` | PK 扫描 | < 5ms |
| `COUNT(tags)` | PK 扫描 | < 3ms |
| `COUNT(post_tags)` | PK 扫描 | < 5ms |
| `SUM(file_size)` | 全表扫描 | < 10ms |
| `GROUP BY source_site` | `source_site` 已有 enum 索引 | < 5ms |
| `GROUP BY rating` | `rating` 已有索引（post.py:62） | < 5ms |
| `TOP 10 tags ORDER BY post_count` | `post_count` 已有索引（tag.py:33） | < 5ms |
| `LIMIT 6 ORDER BY created_at DESC` | 需要 created_at 索引 | < 5ms |

**风险点**：`posts.created_at` 当前**未显式建索引**。10k 行内 PG 可以 seq scan 接受，100k+ 会变慢（参考 [roadmap.md#待做功能](file:///d:/libraries/doc/projects/kura-booru-next/docs/roadmap.md) 中 `random_post` 深分页优化也是同一性能问题）。

**对策**：
- **v0.6.3 范围**：不建索引。仪表盘 admin 专属低频访问，10k-50k 行 seq scan 仍 < 30ms。
- **后续**：如出现 100k+ 帖子，建 `CREATE INDEX ix_posts_created_at ON posts (created_at DESC)`（Alembic 迁移 008）。

### 3.3 缓存策略

**明确不缓存**：
- 仪表盘 admin 专属，访问频率极低（每天几次手动打开）
- 加 Redis 缓存需处理「新增作品后失效」逻辑，得不偿失
- 端点返回 `Cache-Control: private, no-store`（继承自 [main.py#L48-L53](file:///d:/libraries/doc/projects/kura-booru-next/backend/app/main.py#L48-L53) 的现有中间件）

### 3.4 可见性约束

| 数据 | admin | 匿名 |

| 全部 8 个指标 | ✅ | ❌（端点本身 401/403） |
| 端点路径 | `/api/admin/dashboard` | — |

### 问题 1：post_count 累加的 `moved` 是「不重复的」而不是「全部」

**复现场景**（你之前遇到的情况）：

- 三个 tag 指向同一含义：「hatsune_miku」「miku」「初音ミク」
- 假设都有 5 张关联帖子，但其中 4 张是同一批帖子（同一张图同时被打这三个 tag）
- 数据库视角：3 个 tag × 5 个 post_tag 关联 = 15 行，但实际只对应 5 个唯一 post

**执行 A→B 合并**（hatsune_miku 5 关联 → 合并进 miku）：

```
miku 原 post_count = 5
hatsune_miku 5 个关联中：
  - 4 个：post_id 在 miku 里已存在 → skipped +1，删除
  - 1 个：post_id 在 miku 里没有 → moved +1，关联转移到 miku
```

结果：
- `target.post_count += moved` = 5 + 1 = **6** ✅ 正确
- 但 hatsune_miku 的 5 个 post_tag 中只有 1 个真正搬过去，4 个被删除
- `moved + skipped = 5` ✅ 总数对

**再执行 B→C 合并**（miku 6 关联 → 合并进 初音ミク）：

```
初音ミク 原 post_count = 5
miku 6 个关联中：
  - 5 个：post_id 在 初音ミク 里已存在 → skipped +1
  - 1 个：post_id 在 初音ミク 里没有 → moved +1
```

结果：
- `target.post_count += moved` = 5 + 1 = **6**
- 但实际上 初音ミク 现在关联的 post 应该是 **5 个**（miku 的 6 个里有 5 个已存在，只新增 1 个，1 个重复被删）

**问题**：miku 的 `post_count` = 6 包含了「被删除的重复项」的真实计数，而 `moved` 只算了 1 个新增。所以 +1 是对的，但**直觉上 post_count 应该等于唯一 post 数量**。

更严重的场景——你遇到的那个：

**复现场景**：
- 3 个 tag：A=5 张、B=5 张、C=5 张
- 全部都是同一批 5 张图
- A.post_count = 5（实际对应 5 个唯一 post）

**合并 A→B**：
- A 的 5 个 post_tag → B 已有全部 → skipped = 5，moved = 0
- `B.post_count = 5 + 0 = 5` ✅ 看起来对
- 但 A 被删了，B 仍然对应 5 个 post（而不是 0 个？）

**等等**，B 原本就关联这 5 个 post，所以 post_count = 5 是对的。

让我重新看你的描述：「**合入的标签-图片数量没对应增加**」。

### 问题 2：post_count 没有「重新计算」机制

实际场景可能：
- A.post_count = 5（DB 字段）
- 但 A 的实际 post_tags 关联数 ≠ 5（因为历史 bug 漂移或并发更新）
- 合并后 B.post_count = B_old + 0 = 还是 5
- 实际 B 的关联数没变

**根因**：现有实现**完全信任** `moved` 计数，而 moved 只是「这一轮处理中的非重复数」，没有验证最终一致性。

### 问题 3：source.post_count = 0 但 DB 真实值可能不是 0

```python

```

`post_count = 0` 实际上**没用**——因为接下来 `db.delete(source)` 把整行删了，post_count 字段随之消失。这一行是**死代码**，可能让 reviewer 误以为有清理逻辑。

### 问题 4：循环中 N+1 查询

`for pt in source_post_tags` 内部每条都 `await db.execute(exists_stmt)`，如果有 1000 个关联 = 1001 次 SQL。

## A.3 重写实现

修改 [admin_tags.py](file:///d:/libraries/doc/projects/kura-booru-next/backend/app/api/admin_tags.py) 的 `merge_tags` 端点（**整段替换** L164-L223）：

```python
@router.post("/merge", response_model=TagMergeResponse)
async def merge_tags(
    body: TagMergeRequest,
    db: AsyncSession = Depends(get_db),
    _admin: Admin = Depends(get_current_admin),
):
    """Merge source tag into target tag with verified post_count consistency.

    Algorithm:
      1. Load both tags (404 if missing)
      2. Reject self-merge (400)
      3. Bulk-fetch source.post_tags and target.post_tags (1+1 queries)
      4. Compute set difference: posts to move vs posts to delete
      5. DELETE PostTag rows for skipped (already in target)
      6. UPDATE PostTag.tag_id for moved (bulk)
      7. Recompute target.post_count via COUNT(*) — single source of truth
      8. DELETE source tag
      9. Single commit (atomic)

    Returns verified counts so the UI can show a confirmation toast.
    """
    if body.source_tag_id == body.target_tag_id:
        raise HTTPException(
            status_code=400,
            detail="源标签和目标标签不能相同",
        )

    # ── 1. Load tags (with row-level lock to prevent concurrent merges) ──
    source = await db.get(
        Tag, body.source_tag_id, with_for_update=True
    )
    target = await db.get(
        Tag, body.target_tag_id, with_for_update=True
    )

    if source is None:
        raise HTTPException(
            status_code=404,
            detail=f"源标签不存在: {body.source_tag_id}",
        )
    if target is None:
        raise HTTPException(
            status_code=404,
            detail=f"目标标签不存在: {body.target_tag_id}",
        )

    # ── 2. Bulk-fetch post_ids (2 queries, no N+1) ──
    source_post_ids_stmt = select(PostTag.post_id).where(
        PostTag.tag_id == source.id
    )
    target_post_ids_stmt = select(PostTag.post_id).where(
        PostTag.tag_id == target.id
    )

    source_post_ids = set(
        (await db.execute(source_post_ids_stmt)).scalars().all()
    )
    target_post_ids = set(
        (await db.execute(target_post_ids_stmt)).scalars().all()
    )

    # ── 3. Compute set difference ──
    to_move_ids = source_post_ids - target_post_ids   # need to be added
    to_delete_ids = source_post_ids & target_post_ids  # already in target, drop

    # ── 4. Apply changes ──
    if to_delete_ids:
        await db.execute(
            delete(PostTag).where(
                PostTag.tag_id == source.id,
                PostTag.post_id.in_(to_delete_ids),
            )
        )

    if to_move_ids:
        await db.execute(
            update(PostTag)
            .where(
                PostTag.tag_id == source.id,
                PostTag.post_id.in_(to_move_ids),
            )
            .values(tag_id=target.id)
        )

    # ── 5. Recompute target.post_count (single source of truth) ──
    new_count_stmt = select(func.count()).select_from(PostTag).where(
        PostTag.tag_id == target.id
    )
    target_new_count = (await db.execute(new_count_stmt)).scalar() or 0
    target.post_count = target_new_count

    # ── 6. Delete source tag (post_count field is removed along with row) ──
    await db.delete(source)

    # ── 7. Single atomic commit ──
    try:
        await db.commit()
    except Exception as e:
        await db.rollback()
        logger.exception("merge_tags failed: source=%s target=%s", source.id, target.id)
        raise HTTPException(
            status_code=500,
            detail=f"合并失败，事务已回滚: {e}",
        )

    return TagMergeResponse(
        merged=True,
        source_tag_id=source.id,
        source_tag_name=source.name,
        target_tag_id=target.id,
        target_tag_name=target.name,
        posts_moved=len(to_move_ids),
        posts_skipped=len(to_delete_ids),
        target_new_post_count=target_new_count,
        target_old_post_count=len(target_post_ids),
    )
```

### B.3.3 交互脚本（追加到 admin/index.astro 末尾的 `<script>` 中，**纯 ES5** 符合 [CLAUDE.md#Extension content scripts](file:///d:/libraries/doc/projects/kura-booru-next/CLAUDE.md) 约束）

```javascript
// ── 标签 ID 复制浮泡 ──
(function() {
  var tooltip = document.getElementById('tag-id-tooltip');
  var tooltipText = document.getElementById('tag-id-tooltip-text');
  var copyBtn = document.getElementById('tag-id-copy-btn');
  if (!tooltip || !tooltipText || !copyBtn) return;

  var currentTagId = null;
  var showTimer = null;
  var hideTimer = null;

  function showTooltip(target) {
    var rect = target.getBoundingClientRect();
    currentTagId = target.getAttribute('data-tag-id');
    tooltipText.textContent = currentTagId;
    copyBtn.textContent = '复制';
    copyBtn.disabled = false;

    // 定位在元素上方居中
    tooltip.style.left = (rect.left + rect.width / 2) + 'px';
    tooltip.style.top = (rect.top - 8) + 'px';
    tooltip.style.transform = 'translate(-50%, -100%)';
    tooltip.classList.remove('hidden');
  }

  function hideTooltip() {
    tooltip.classList.add('hidden');
    currentTagId = null;
  }

  function scheduleShow(target) {
    clearTimeout(hideTimer);
    clearTimeout(showTimer);
    showTimer = setTimeout(function() { showTooltip(target); }, 200);
  }

  function scheduleHide() {
    clearTimeout(showTimer);
    hideTimer = setTimeout(hideTooltip, 200);
  }

  // 委托到所有 .tag-id-cell（动态插入也生效）
  document.addEventListener('mouseover', function(e) {
    var cell = e.target.closest && e.target.closest('.tag-id-cell');
    if (cell) scheduleShow(cell);
  });
  document.addEventListener('mouseout', function(e) {
    var cell = e.target.closest && e.target.closest('.tag-id-cell');
    if (cell) scheduleHide();
  });

  // 复制按钮
  copyBtn.addEventListener('click', function() {
    if (!currentTagId) return;
    var done = function() {
      copyBtn.textContent = '✓ 已复制';
      copyBtn.disabled = true;
      setTimeout(function() {
        copyBtn.textContent = '复制';
        copyBtn.disabled = false;
      }, 1500);
    };
    var fail = function() {
      copyBtn.textContent = '✗ 失败';
      setTimeout(function() { copyBtn.textContent = '复制'; }, 1500);
    };
    if (navigator.clipboard && navigator.clipboard.writeText) {
      navigator.clipboard.writeText(currentTagId).then(done, fail);
    } else {
      // Fallback: 临时 textarea + execCommand
      try {
        var ta = document.createElement('textarea');
        ta.value = currentTagId;
        ta.style.position = 'fixed';
        ta.style.opacity = '0';
        document.body.appendChild(ta);
        ta.select();
        document.execCommand('copy');
        document.body.removeChild(ta);
        done();
      } catch (e) {
        fail();
      }
    }
  });

  // 浮泡自身 mouseleave 隐藏（让用户能 hover 到浮泡按钮上）
  tooltip.addEventListener('mouseenter', function() {
    clearTimeout(hideTimer);
  });
  tooltip.addEventListener('mouseleave', function() {
    scheduleHide();
  });
})();
```

### B.3.4 行为表

| 场景 | 行为 |
|---|---|
| hover 单元格 | 200ms 后浮泡出现在元素上方居中 |
| 移出单元格 | 200ms 延迟后隐藏（给用户时间移到浮泡上） |
| hover 浮泡 | 保持显示，clear hide timer |
| 点击「复制」 | 写入剪贴板，按钮变「✓ 已复制」1.5s |
| 旧浏览器无 Clipboard API | 降级到 textarea + execCommand |
| 极端情况 | 浮泡内 `<code>` 仍可手动全选（`select-all`） |

### B.3.5 移动端

`mouseover` / `mouseout` 在移动端不触发（无 hover 概念）。建议：

- 移动端用 `tap` 触发：`e.target.closest('.tag-id-cell')` + 监听 `click`
- 浮泡内容自动包含「复制按钮」，tap 单元格 → 浮泡出现 → tap 按钮复制

简化为对移动端和桌面端都用 click 触发：

```javascript
// 桌面 hover
document.addEventListener('mouseover', function(e) {
  var cell = e.target.closest && e.target.closest('.tag-id-cell');
  if (cell) scheduleShow(cell);
});
// 移动 tap
document.addEventListener('click', function(e) {
  var cell = e.target.closest && e.target.closest('.tag-id-cell');
  if (cell) {
    // 如果点击的就是 .tag-id-cell 本身，切换浮泡可见性
    if (currentTagId === cell.getAttribute('data-tag-id') && !tooltip.classList.contains('hidden')) {
      hideTooltip();
    } else {
      showTooltip(cell);
    }
  }
});
```
