## 二、功能 3：Pixiv 多图——只抓第一张 ✅ 已完成

### 问题根因

`backend/app/services/gallery_dl.py:_download_sync`：

- `DataJob` 部分**正确**——遍历 items 时遇到第一个 `Message.Url` 即 `break`，所以 `result["image_urls"]` 只含第一张、metadata（title/tags/description）也是第一张的。
- **`DownloadJob` 是 bug 源头**：对多图 Pixiv 帖子，gallery-dl 的 DownloadJob 会下载**所有**图片到 tmpdir。原代码 `os.walk` 遍历所有文件，`downloaded_path` 被反复覆盖，最终 `image_bytes` 指向 **walk 到的最后一个文件**（顺序不确定，可能与 metadata 的第一张不一致）。
- **后果**：存储的图片可能是多图帖里**任意一张**，且不一定匹配 title/tags 所在的第一张 → 数据不一致。

### 修复（`backend/app/services/gallery_dl.py`）

1. `setup_gallery_dl_config()` 新增 Pixiv 专属配置（放在 Pixiv auth 之后、rate-limit 之前）：
   ```python
   # Multi-image Pixiv posts: only grab the first image.
   gallery_dl.config.set(("extractor", "pixiv"), "image-range", "1-1")
   ```
   `image-range` 是 gallery-dl 官方标准选项（见 `docs/configuration.rst`），`"1-1"` 表示只下载第 1 张。**仅设 pixiv extractor**，不动其他来源。

2. **防御性加固** `_download_sync` 的文件读取逻辑：不再 `os.walk` 取最后一个，改为收集所有非 `.json` 文件并**排序后取第一个**。gallery-dl 多图文件名含 `_p0/_p1` 页码序号，排序即第一张。配合 `image-range` 后只会有一个文件，双重保险：
   ```python
   candidates = sorted(
       os.path.join(root, fname)
       for root, _dirs, files in os.walk(tmpdir)
       for fname in files
       if not fname.endswith(".json")
   )
   if candidates:
       downloaded_path = candidates[0]
   ```

- 不涉及：`process_image.py` 的 `image_urls` 去重逻辑保持（防御 infojson 重复项，与多图无关）。
- 语法校验通过（`python -m py_compile`）。

## 四、功能 1：网页端批量导入队列实时更新（SSE）⏳ 进行中

### 问题

`import.astro` 现在只显示「已入队 ✓ / 失败 ✗」，任务在 ARQ worker 后台跑完后页面无任何反馈。Bot 用 `poll_job_result`（ARQ `job.result()`）感知完成——网页端要对齐这个体验。

### 方案：SSE 单端点，复用现有 ARQ 轮询

**为什么 SSE 而非 WebSocket**：单向推送、纯 HTTP、浏览器自动重连、FastAPI `StreamingResponse` 原生支持、零新依赖。与 Bot 的「poll + notify」模型一一对应。

### 4.1 后端 SSE 端点 ✅ 已完成（`backend/app/api/tasks.py`）

新增 `GET /api/tasks/web-import/stream`：

- **认证**：`Depends(get_current_admin)`（与 `web-import` 一致；SSE 必须管理员，否则会泄漏非安全任务结果）。
- **入参**：query `task_ids: str`（逗号分隔的 ARQ job ID 列表，上限 50）。
- **轮询逻辑**（对齐 Bot `poll_job_result` + `_poll_and_notify`）：
  - 用本地 `_parse_redis_url()`（与 worker/bot 同款解析）建 ARQ pool。
  - 每个 task_id 构造 `Job(task_id, redis=pool)`。
  - 循环（最长 **5 分钟**，每 **2s** 一轮）：对所有未完成 job 调 `await job.status()`：
    - `JobStatus.complete` → `result_info()` 取结果分类。
    - `JobStatus.not_found` → 任务丢失，按失败处理。
    - `in_progress` / `queued` / `deferred` → 继续等待。
  - 每轮 yield `: ping\n\n` 心跳（保活）。
  - 全部完成或超时后发 `done`，finally 关闭 pool。
- **事件协议**（SSE `event:` + JSON `data:`）：
  - `event: progress` — `{url_index, task_id, status, detail, post_id?|existing_post_id?}`
  - `event: done` — `{total, succeeded, failed, timed_out}`
  - `event: error` — `{message}`
- **结果分类**（复刻 Bot `_poll_and_notify` 文案）：`success`→✅ 导入成功（含 `post_id`）、`duplicate`→⚠️ 重复图片（含 `existing_post_id`）、`too_large`→⚠️ 图片过大、`failed`→❌ 下载失败/处理失败。
- **响应头**：`media_type="text/event-stream"`、`Cache-Control: no-cache`、`Connection: keep-alive`、`X-Accel-Buffering: no`。
- **arq API 兼容性**：`JobStatus` 正确取值为 `complete`（非 `done`）、`not_found`、`in_progress`、`queued`、`deferred`（**无 `aborted`/`done`**，已对照 arq 官方源码修正）。`result_info()` 用 `getattr(result_info, "result", None)` 鸭子类型访问，兼容多版本。
- 语法校验通过。

## 六、风险与权衡

- **SSE 超时**：ARQ 单 job 上限 5min、max_tries=3；SSE 流设 5min 上限，超时未完成任务在前端显示「处理超时」，`done` 事件带 `timed_out` 计数。
- **Caddy 缓冲**：必须加 `flush_interval -1`，否则 SSE 在 Caddy 后完全失效（最易踩的坑）。
- **image-range 兼容性**：已在 gallery-dl 官方 configuration.rst 确认 `extractor.<name>.image-range` 为标准选项。若个别版本不识别，防御性「取第一个文件」逻辑兜底。
- 不引入新依赖、不改 DB schema、不改 Post 模型。
