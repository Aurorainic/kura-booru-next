## B. 搜索框修复 (S1 + S3)

### B.1 修复 S1: 乱序响应覆盖

**根因**: `app/components/SearchBar.vue:38-50` 的 debounce 只能合并请求发起，不能取消。慢请求 A 可能在快请求 B 之后返回 → 旧结果覆盖新结果。

**修复**: fetchAutocomplete 加 `AbortController`，组件侧每次新输入或 unmount 时 `abort()`。

**改动文件**:
- `app/composables/api.ts` — `fetchApi` 接 `signal: AbortSignal`，向下传 `fetch()`
- `app/composables/api.ts` — `fetchAutocomplete(prefix, signal?)`
- `app/components/SearchBar.vue` — `onInput` 维护 `currentController`，新输入 `abort()` 旧的；`onUnmounted` 兜底 abort

**预计 diff**: 8-12 行

**验证**: 手测：输入 `abc`、立即输入 `abcd`，网络被 throttle 模拟 A 慢 B 快，确认 UI 显示 B 的结果不是 A 的；unmount 时控制台无 "setState on unmounted" 警告。

### B.2 修复 S3: CJK Ext B 被砍

**根因**: `server/utils/search/suggest.ts:81` 用 `[^\w一-龥-]` 正则，JS `\w` 只匹配 ASCII，`一-龥` 到 BMP 末尾。CJK Ext B (U+20000+) 等被砍。

**修复**: 一行正则替换
```ts
// before
const safe = String(prefix || '').replace(/[^\w一-龥-]/g, '').slice(0, 64)
// after
const safe = String(prefix || '').replace(/[^\p{L}\p{N}]/gu, '').slice(0, 64)
```

**改动文件**: `server/utils/search/suggest.ts:81` 一行

**预计 diff**: 1 行

**验证**: 用罕用字前缀（如 "𠮷野"）搜索，不被砍光。

### B.3 跳过的搜索框问题

- S4-S6 (UX 改善) — 留 backlog

## C. Pixiv 单帖多图能力 (LAI-24)

**核心改动**:
1. **DB schema**: `posts` 加 `series_id` (uuid, nullable) + `page_index` (integer, nullable)，`source_id` 唯一约束改成 `(source_site, source_id, page_index)` 复合唯一
2. **Sidecar**: `image-range` 从 `1-1` 改为 `1-9999`，对同一 `illust_id` 产出 N 个 job
3. **Pipeline**: 接 `page_index` 字段，写入 post；同一 illust 的第一个 post 分配新 `series_id`，后续 N-1 个 post 共享同一 `series_id`
4. **API**: `/api/posts/by-source` 行为改为返回 series 下所有图；`/api/posts/[id]` 详情页附加 `series` 字段（同 series 的 N 个 post 简要信息）
5. **前端**: 详情页顶部在多图时显示 "1/3" + 缩略图带 + 左右键切换

### C.1 DB schema 迁移

新文件 `drizzle/0004_post_series.sql`:
```sql
ALTER TABLE posts
  ADD COLUMN series_id uuid,
  ADD COLUMN page_index integer,
  ADD COLUMN page_count integer;  -- 冗余: 该 series 总页数, 写时填, 读时不需要 join

-- 旧唯一索引 (source_site, source_id) 是单图假设, drop & replace
DROP INDEX IF EXISTS ix_posts_source_site_id;
CREATE UNIQUE INDEX ix_posts_source_site_id_page
  ON posts (source_site, source_id, page_index);

-- 同 series 查询用
CREATE INDEX ix_posts_series_id ON posts (series_id)
  WHERE series_id IS NOT NULL;
```

**风险**: 现有 604 个 post 都没有 `series_id` / `page_index` / `page_count`，需要 backfill：每个 post 视为"单图 series"，填 `series_id = id`（自引用）、`page_index = 1`、`page_count = 1`。Backfill 一次性脚本：
```sql
UPDATE posts
SET series_id = id, page_index = 1, page_count = 1
WHERE series_id IS NULL;
```
**回滚**: 删列 + 重建原索引即可；新数据无影响。

**预计 diff**: schema 1 个新文件 + backfill 一次性 SQL + `drizzle.config.ts` 不变（drizzle-kit 会自动生成）

### C.2 Sidecar 改动

**文件**: `sidecar/sidecar.py`

**关键改动**:
1. 删 `image-range=1-1`，改 `image-range=1-9999`（行 107）
2. `download_with_gallery_dl(url)` 改返回 `list[tuple[bytes, dict]]` 而不是单个 — 一帖 N 张图就是 N 个
3. 新 metadata 字段: `page_index` (1-based), `page_count` (总数), `series_source_id` (= source_id 同一值)
4. `process_job` 一次入队 N 个 result 到 `kura:pending_results`，**每个 result 仍走独立 job_id 走原 pipeline**（不改 pipeline 内部逻辑，只在 metadata 里带 page 信息）
5. 限制最大 N: `image-range=1-32`（Pixiv 漫画/动图一般 30 页内是上限，32 是防爆库安全值；超过 32 的 illust 罕见）
6. 对 Ugoira 走原 1 张图路径不变（detect: `data["type"] == "ugoira"`）

**预计 diff**: 30-50 行（含 metadata 抽取 + 循环）

**关键决定 — 走 jobs 还是单 job 拆 N**:
- 方案 1: 1 个 URL 入 1 个 job，sidecar 一次下载 N 张，**把 N 张图直接 base64 塞进 1 个 result，pipeline 拆开 insert** — 简单但 result 可能超大（每张图 ~1-10MB base64）
- 方案 2: 1 个 URL 入 1 个 job，sidecar 一次下载 N 张，**回写 N 个独立 result 到 kura:pending_results**，N 个独立 pipeline 处理 — 单个 result 不大，复用现有 pipeline 路径
- **选 方案 2** — 与现有架构一致，无新路径，pipeline worker 几乎无改动

### C.3 Pipeline 改动

**文件**: `server/utils/pipeline.ts`

**关键改动**:
1. 接 `meta.page_index` / `meta.page_count` / `meta.series_source_id`
2. **Series 协调逻辑**（在 dedup 之后、insert 之前）:
   - 查 `posts` 表: `SELECT id, page_count FROM posts WHERE source_site=? AND source_id=? AND page_index IS NOT NULL ORDER BY page_index LIMIT 1` 拿"系列首个 post"
   - 若无: 当前是第 1 张 → 先插入当前 post，拿到 postId，把它当 series_id 回填自己，再 update 自己的 page_count = meta.page_count
   - 若有: 当前是第 N 张 → 拿 series_id，复用，page_index 用 meta 的
3. **去重** 也要适配: 同 series 内不同 page_index 不应互相 dedup（phash 不同自然不会 dedup 同图，但跨 page 误判是 phash 算法问题不该在这里管）
4. **Phash dedup 跨 series 不变** — 同图上传两次（不同 URL 不同时间）仍能 dedup，靠 phash 兜底
5. auto-rating / tag 关联：所有同 series 的 post 共享 tags + rating（auto-rating 用同一 tag 集合算一次，所有 page 同一结果）

**预计 diff**: 30-40 行

**并发竞争**: 两个 worker 同时处理同一 series 的不同 page → 都要"找 series 首个 post"，可能都拿到空 → 都"自己是首个"。需要：
- 加 DB 层 `(source_site, source_id, page_index)` 唯一约束（已在 C.1）
- insert 时 catch unique violation → 回退到"复用已存在"分支
- 简单可靠，不引分布式锁

### C.4 API 改动

**文件**: `server/routes/api/posts/[id].get.ts` + `server/utils/queries.ts`

**关键改动**:

### E.1 鉴权策略（待用户确认 #5）

三个候选方案，**首要任务是选出其中之一**：

| 方案 | 描述 | UX | 安全 | 改动量 |
|---|---|---|---|---|
| **a. 用户名/密码 + Cookie** | popup 让用户填 admin 账号密码，service-worker 调 `POST /api/auth/login` 拿 cookie，存 `chrome.storage.session`（非 sync）；之后每次请求带 cookie | 多填两个字段（4 个） | 用户级鉴权（部署者权限风险最低） | 最小：无需后端新增端点，只需 extension 改登录逻辑 |
| **b. 专用 Extension API Key** | admin 后台生成/吊销；存 DB（`extension_keys` 表）；service-worker 发送 `X-Api-Key: <key>` + 中间件 `checkExtensionKey`；**不复用 `BACKEND_API_KEY`** | 单一字段 | 可吊销、可追溯、最干净 | 中：1 表 + 1 中间件 + admin UI + popup 改造 |
| **c. 直接复用 `BACKEND_API_KEY`** | 现在就是这样——**强烈不推荐**：把部署者级密钥下发给所有用户 | 单字段 | ❌ 全用户共享部署者权限，吊销=全员不可用 | 0 |

**推荐 b**：admin 可吊销单用户、审计每 key 用量、吊销不波及其他用户；缺点是首次需要去 admin 后台生成 key。但 popup 已经有"获取 key"的指引位（截图/扫码可以放到文档）。

**若选 a**：后端零改动。extension 直接调 `/api/auth/login` 拿 cookie。

### E.2 后端改动（基于方案 b）

> 若选 a：跳过此节，extension 只改调用方式即可。

新增一个表 `extension_keys`（不需要 Drizzle 迁移脚本，手写在 `server/utils/schema.ts`）：

```ts
export const extensionKeys = pgTable('extension_keys', {
  id: uuid('id').primaryKey().defaultRandom(),
  name: varchar('name', { length: 64 }).notNull(),  // 友好名，比如 "Lainsaka Chrome"
  keyHash: varchar('key_hash', { length: 64 }).notNull().unique(),  // sha256(rawKey)
  keyPrefix: varchar('key_prefix', { length: 8 }).notNull(),  // "kb_ext_xxxx" 前 8 位用于 UI 识别
  createdBy: text('created_by').notNull(),  // admin username（创建者）
  createdAt: timestamp('created_at').defaultNow().notNull(),
  lastUsedAt: timestamp('last_used_at'),
  revokedAt: timestamp('revoked_at'),  // null = active
})
```

- `rawKey` 格式：`kb_ext_` + 32 字符 base62；**只创建时返回给 admin 一次**，DB 只存 hash（参照 GitHub PAT 模型）
- 中间件 `server/middleware/04-extension-auth.ts`：检测 `X-Api-Key` 前缀是 `kb_ext_` 则走 `checkExtensionKey`（不与 `BACKEND_API_KEY` 冲突）；命中后在 `event.context.extensionKey = { id, name }`
- 修改 `server/routes/api/tasks/web-import.post.ts`：同时支持 admin session 和 extensionKey；rate limit per key（如 60/min）独立于 `BACKEND_API_KEY` 的 30/min

### E.4 "类型选择"具体定义（待用户确认 #6）

"类型选择"在 popup 里最自然的是 **rating override**：
- 用户想手动标注这是 R-18 还是 safe，不想被 AI 误判（Pixiv 的 AI 自动 rating 不准是已知问题）
- 选项：`自动` / `safe（公开）` / `questionable（敏感）` / `explicit（限制）`
- 后端 `web-import` 增加可选 `force_rating` 字段；存在时跳过 AI auto-rating

**若用户原意不是 rating**，可能候选：
- **是否启用 AI 标签处理**（开/关，对应 `reprocessTagsAPI`）
- **是否启用自动 rating**（开/关）
- **是否启用 series grouping**（对 PR-C 多图支持时，按 series 全量抓 vs 只抓第一页）
